## Context

학지사 도서 카탈로그를 키워드로 검색하고 같은 분야 도서를 추천하는 MVP다. 목표는 정식 검색·추천 인프라를 만들기 전에 핵심 흐름(검색 → 상세 → 관련 도서)을 빠르게 검증하는 것이다.

- **스택**: Next.js(App Router) + Supabase(Postgres).
- **데이터**: 초기엔 수동 시드(50~100권). 정식 카탈로그 연동은 비목표.
- **제약**: 가볍게/빠르게. 검색·추천 모두 추가 인프라(검색엔진, ML) 없이 Postgres 안에서 해결한다.
- **이해관계자**: 도서를 찾는 일반 사용자(학생·실무자), MVP를 검증하는 제작자.

## Goals / Non-Goals

**Goals:**
- 키워드 검색, 분야별 둘러보기, 도서 상세, 같은 분야 추천을 단일 Postgres 테이블로 구현.
- 콜드스타트(행동 데이터 0) 상태에서도 첫날부터 작동하는 추천.
- 향후 검색 정밀도 업그레이드(`pg_bigm`/pgvector)를 스키마 변경 없이 끼워 넣을 수 있는 구조.

**Non-Goals:**
- 로그인/회원, 행동 로그, 협업필터링.
- 벡터/의미 검색, 형태소 분석.
- 결제·장바구니, 관리자 CMS, 실시간 크롤링.

## Decisions

### 결정 1: 추천은 "같은 분야" 규칙 기반 (벡터/ML 아님)
- **선택**: 상세 페이지에서 `category`가 같은 다른 도서를 최대 5권 노출.
- **이유**: 행동 데이터가 없는 콜드스타트에서도 카탈로그만으로 첫날부터 작동한다. SQL 한 줄(`WHERE category = ? AND id <> ? LIMIT 5`)로 끝나 운영 부담이 없다.
- **대안**: (a) 협업필터링 — 클릭/구매 이력 필요, 첫날 작동 불가. (b) pgvector 콘텐츠 유사도 — 임베딩 파이프라인·비용 추가. MVP 가치 대비 과함.
- **확장 경로**: 추천 품질이 아쉬우면 동일 인터페이스("관련 도서 N권")를 유지한 채 내부 쿼리만 키워드 겹침 → 벡터 유사도로 교체.

### 결정 2: 검색은 `ILIKE` 부분일치 (전문검색/외부엔진 아님)
- **선택**: `title`, `author`, `description`에 대해 `ILIKE '%키워드%'` (대소문자 무시).
- **이유**: Postgres 기본 한국어 전문검색(`tsvector`)은 형태소 분석이 없어 "상담심리" → "상담 심리학" 같은 매칭이 약하다. 외부 검색엔진(Meilisearch/Typesense)은 미니 스코프에 과함. 소규모 카탈로그(수백 권)에선 `ILIKE`가 충분히 빠르고 예측 가능하다.
- **대안**: (a) `tsvector` FTS — 한국어 정밀도 한계. (b) `pg_bigm` n-gram — 가능하지만 확장 설치/인덱스 운영 추가. (c) 외부 엔진 — 인프라 과다.
- **확장 경로**: 카탈로그가 커지거나 정밀도가 필요하면 `pg_bigm` GIN 인덱스 또는 pgvector로 교체(스키마 호환).

### 결정 3: 단일 `contents` 테이블 + 정규화 없음
- **선택**: 분야(category)를 별도 테이블로 빼지 않고 `contents.category` 텍스트 컬럼으로 둔다.
- **이유**: 분야 수가 적고 고정적이며, 둘러보기는 `SELECT DISTINCT category`로 충분하다. 조인을 없애 쿼리·시드가 단순해진다.
- **대안**: `categories` 테이블 + FK — 정합성은 좋아지나 MVP엔 불필요한 복잡도.

### 결정 4: 읽기 전용 데이터 접근 (Supabase RLS public select)
- **선택**: `contents`에 public `SELECT`만 허용하는 RLS 정책. 쓰기는 시드 적재 시 secret 키(서버 전용)로만.
- **이유**: 익명 사용자에게 검색·조회만 필요하다. publishable 키가 노출돼도 읽기 외 위험이 없다.
- **대안**: RLS 비활성화 — 빠르지만 실수로 쓰기 노출 가능. 최소 권한 원칙에 어긋남.

### 데이터 흐름

```
[홈/검색창] --검색어--> Next 서버(쿼리) --ILIKE--> Supabase.contents --> [검색 결과]
[검색 결과] --도서 선택--> [상세] --(category 동일, id 제외, LIMIT 5)--> [같은 분야 추천]
```

## Risks / Trade-offs

- **한국어 검색 정밀도** → `ILIKE`는 부분 문자열만 잡아 어순/형태소 변형에 약함. 완화: MVP 검증 목적엔 수용, 정밀도 필요 시 `pg_bigm`/벡터로 교체(결정 2 확장 경로).
- **카탈로그 성장 시 `ILIKE` 성능 저하** → 수만 건 이상에서 풀스캔 비용 증가. 완화: 현재 규모(수백 권)에선 무의미. 커지면 trigram(`pg_trgm`) GIN 인덱스 추가.
- **추천 다양성 부족** → 같은 분야만 보여줘 동질적일 수 있음. 완화: 정렬에 약간의 무작위/최신순 가미, 추후 키워드 겹침으로 확장.
- **시드 데이터 품질·저작권** → 표지/소개를 수동 수집할 때 출처·권리 확인 필요. 완화: 학지사 제공 데이터 우선, 표지 없는 항목 허용(스펙에 반영).

## Migration Plan

신규 프로젝트라 마이그레이션 대상은 없다. 배포 절차:
1. Supabase에 `contents` 테이블 + RLS(public select) 생성.
2. 시드 JSON/CSV 적재(secret 키, 서버 전용).
3. Next.js 앱에 publishable 키 환경변수 연결 후 배포(예: Vercel).
- **롤백**: 앱 배포 되돌리기 + 필요 시 `contents` truncate/재적재. 사용자 데이터가 없어 롤백 위험 낮음.
